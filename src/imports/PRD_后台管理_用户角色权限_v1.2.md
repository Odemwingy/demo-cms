## 1. 背景与目标

### 1.1 背景

地面内容管理平台（EasyCMS）当前处于 MVP 阶段，平台服务两类用户：

- **内部员工**：包括内容运营、审核人员、发布人员、配置管理员等，负责内容准备、审核签核、周期发布和配置维护
- **企业客户（内容服务商）**：以组织为单位接入平台，负责按分配任务批量导入内容、处理素材，仅可访问分配给本组织的数据范围

两类用户共用同一平台，通过角色和数据范围权限隔离操作边界。当前缺乏统一的账号与权限管控体系，存在以下问题：

- 无法按岗位和组织限制操作范围，存在越权访问和误操作风险
- 无法追溯关键操作行为，出现问题难以定责
- 账号开通和权限调整依赖人工沟通，效率低

### 1.2 目标

- 建立统一的用户账号生命周期管理能力
- 通过角色抽象降低逐用户授权成本
- 支持页面级 + 操作级权限控制，保证最小权限原则
- 对关键操作形成完整审计日志，满足追责与排障需求

### 1.3 不做什么（Out of Scope）

- 忘记密码 / 邮件激活（首期由管理员重置密码，二期补充）
- 组织管理页面（首期预置枚举，二期补充）
- 数据范围权限过滤逻辑（首期预留字段，默认全量，二期补充）
- 互斥角色约束（二期补充）
- 批量导入用户（二期补充）
- 消息中心（二期补充）
- 通知配置（二期补充）
- 系统参数页（二期补充）

## 3. 功能详细说明

### 3.1 登录与账号安全

**功能描述**

平台采用自建账号密码体系，不对接 SSO/LDAP。首期不做邮件激活和忘记密码流程，账号由系统管理员统一创建和维护。

**登录流程**

1. 用户在登录页输入用户名和密码，点击「登录」
2. 后端校验账号状态、密码正确性
3. 登录成功后跳转工作台首页
4. 首次登录（或管理员重置密码后）强制跳转修改密码页，完成修改后再进入系统
5. 账号处于停用/锁定状态时，提示对应原因，不允许登录

**密码规则**

| 参数项 | 规则 |
|--------|------|
| 密码最小长度 | 8 位 |
| 密码复杂度 | 必须同时包含大写字母、小写字母、数字 |
| 登录失败锁定阈值 | 连续失败 5 次自动锁定账号 |
| 锁定解除方式 | 管理员手动解锁 或 30 分钟后自动解锁 |
| 会话超时 | 8 小时无操作自动退出 |
| 首次登录改密 | 是，强制修改初始密码后方可使用 |

**异常处理**

| 异常场景 | 提示文案 |
|----------|---------|
| 用户名或密码错误 | 「用户名或密码错误，请重新输入」 |
| 账号已停用 | 「账号已停用，请联系管理员」 |
| 账号已锁定 | 「账号已锁定，请联系管理员或 30 分钟后重试」 |
| 会话超时 | 「登录已过期，请重新登录」 |

**验收标准**

- [ ] 用户名密码正确可正常登录
- [ ] 连续失败 5 次后账号锁定，无法继续尝试
- [ ] 首次登录强制跳转修改密码页，未修改不可进入系统
- [ ] 停用/锁定账号登录时展示正确提示
- [ ] 8 小时无操作后会话过期，跳转登录页

### 3.2 用户管理

**功能描述**

系统管理员统一维护平台用户账号，包括创建、编辑、停用、解锁、重置密码等操作。

#### 3.2.1 用户列表页

**页面元素**

- 顶部搜索：关键字搜索（用户名 / 姓名）
- 筛选项：所属组织、用户类型（内部员工 / 企业客户）、角色、账号状态
- 操作按钮：新建用户、批量停用、批量启用
- 列表字段：用户名、姓名、所属组织、角色、账号状态、最近登录时间、创建时间、操作

**操作列**

| 操作 | 可见条件 |
|------|---------|
| 编辑 | 始终显示 |
| 停用 | 账号状态为「启用」时显示 |
| 启用 | 账号状态为「停用」时显示 |
| 解锁 | 账号状态为「锁定」时显示 |
| 重置密码 | 始终显示 |

**分页规则**

- 每页默认展示 20 条，支持切换每页条数（20 / 50 / 100）

#### 3.2.2 新建 / 编辑用户

**核心字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 用户名 | 文本 | 是 | 登录名，全局唯一，创建后不可修改 |
| 姓名 | 文本 | 是 | 展示名称 |
| 用户类型 | 单选 | 是 | 内部员工 / 企业客户；影响默认可选角色范围 |
| 初始密码 | 文本 | 是（新建时） | 用户首次登录强制修改 |
| 所属组织 | 下拉选 | 是 | 首期采用预置枚举或初始化数据维护，暂不提供系统参数页面维护 |
| 所属项目 | 多选 | 否 | 限定用户可访问的项目范围，为空则不限制 |
| 角色 | 多选 | 是 | 可绑定多个角色 |
| 账号状态 | 单选 | 是 | 新建时默认「启用」 |
| 备注 | 文本 | 否 | 补充说明 |

**规则说明**

- 用户名全局唯一，重复时阻止保存并提示「用户名已存在」
- 用户类型为「内部员工」时，可选角色包含所有预置角色；用户类型为「企业客户」时，可选角色仅限内容服务商相关角色，不可分配系统管理员、发布人员等内部角色
- 至少保留一个系统管理员账号处于启用状态，不可停用最后一个管理员
- 用户可绑定多个角色，最终权限取所有角色权限的并集，冲突时以禁止优先
- 账号停用后立即禁止登录，但历史操作记录保留，不影响责任归属追溯

#### 3.2.3 重置密码

1. 管理员点击「重置密码」，弹出确认弹窗
2. 输入新密码（需符合密码规则）或勾选「系统自动生成」
3. 确认后密码重置，用户下次登录强制修改密码
4. 写入审计日志

**验收标准**

- [ ] 用户列表支持搜索、筛选、分页
- [ ] 新建用户时用户名重复校验正确
- [ ] 停用最后一个管理员时系统阻止操作
- [ ] 停用账号后用户无法登录
- [ ] 重置密码后用户下次登录强制改密
- [ ] 所有用户操作写入审计日志

### 3.3 角色管理

**功能描述**

通过角色对平台访问和操作权限进行抽象，降低逐用户授权成本。平台采用 RBAC 模型，所有权限均通过角色分配，不支持直接给用户授权。

#### 3.3.1 系统预置角色

| 角色 | 说明 |
|------|------|
| 系统管理员 | 平台最高权限，负责用户、角色、权限、审计管理 |
| 项目管理员 | 负责项目范围内的用户和日志查看 |
| 内容运营 | 负责内容录入、编辑、分类、状态检查 |
| 内容服务商 | 负责按分配任务批量导入内容、处理素材 |
| 审核人员 | 负责技术审核、全量审核、变更审批 |
| 发布人员 | 负责周期管理、完整性校验、导出发布 |
| 配置管理员 | 负责配置档、分类集、适配规则等配置维护 |
| 只读用户 | 仅查看，不允许任何写操作 |
| 支持排障人员 | 查看配置、内容与导出结果，不允许修改 |

系统预置角色不可删除，可复制为新角色使用。

#### 3.3.2 角色列表页

**页面元素**

- 搜索框：按角色名称搜索
- 筛选项：角色状态（启用/停用）
- 操作按钮：新建角色
- 列表字段：角色名称、角色编码、绑定用户数、状态、最近修改时间、操作

**操作列**：编辑、复制、停用/启用、删除（仅未绑定用户且非系统预置时可删除）

#### 3.3.3 新建 / 编辑角色

**核心字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 角色名称 | 文本 | 是 | 全局唯一 |
| 角色编码 | 文本 | 是 | 机器识别编码，全局唯一，创建后不可修改 |
| 角色说明 | 文本 | 否 | 岗位描述 |
| 数据范围 | 单选 | 是 | 首期默认「全量」，字段预留，逻辑二期实现 |
| 状态 | 单选 | 是 | 启用 / 停用 |

**停用角色规则**

- 停用角色前，系统校验该角色下是否存在状态为「启用」的绑定用户
- 若存在活跃用户，阻止停用，提示「该角色下存在 N 个活跃用户，请先解绑或停用相关用户后再停用角色」，并提供跳转绑定用户列表的快捷入口
- 角色停用按钮 hover 时，展示当前绑定的活跃用户数量

**验收标准**

- [ ] 系统预置角色不可删除，可复制
- [ ] 角色名称/编码重复时阻止保存
- [ ] 停用含活跃用户的角色时系统正确拦截并提示
- [ ] 删除已绑定用户的角色时系统阻止操作
- [ ] 所有角色操作写入审计日志

### 3.4 菜单管理

**功能描述**

菜单管理是权限控制的基础数据层，负责维护平台的完整菜单树结构。角色的页面级权限和操作级权限均以菜单节点为挂载点。系统预置初始菜单树，管理员可在界面上增删改，新模块上线时无需通过 Seeder 或发布代码注册菜单。

**菜单节点类型**

| 类型 | 说明 | 是否有路由 | 是否可挂按钮 |
|------|------|-----------|------------|
| 目录 | 一级导航，无路由，仅作为分组容器 | 否 | 否 |
| 菜单项 | 有对应前端路由的页面节点 | 是 | 是 |
| 按钮 | 挂载在菜单项下的操作级权限节点（如新建、删除、导出） | 否 | — |

> 外链类型首期不做。

#### 3.4.1 菜单列表页

**页面结构**

采用树形表格展示，支持展开/收起，直观呈现层级关系。

**列表字段**

| 字段 | 说明 |
|------|------|
| 菜单名称 | 带层级缩进展示，目录/菜单项/按钮用图标区分 |
| 菜单类型 | 目录 / 菜单项 / 按钮 |
| 权限标识 | 如 `admin:users:list`、`admin:users:create`，角色权限配置时使用 |
| 路由地址 | 前端路由路径，按钮类型不填 |
| 图标 | 仅目录和菜单项有图标（lucide-react 图标名） |
| 排序 | 同级节点间的显示顺序 |
| 状态 | 启用 / 停用 |
| 操作 | 新增子节点、编辑、停用/启用、删除 |

**顶部操作**：新增根节点、展开全部、收起全部

#### 3.4.2 新建 / 编辑菜单节点

**核心字段**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 父节点 | 树选择 | 否 | 为空则为根节点（一级目录） |
| 菜单类型 | 单选 | 是 | 目录 / 菜单项 / 按钮 |
| 菜单名称 | 文本 | 是 | 界面展示名称，最多 20 字 |
| 权限标识 | 文本 | 是 | 全局唯一，格式：`模块:资源:动作`，如 `admin:users:create` |
| 路由地址 | 文本 | 菜单项必填 | 前端路由路径，如 `/admin/users` |
| 组件路径 | 文本 | 菜单项必填 | 前端组件相对路径，如 `features/users/UserListPage` |
| 图标 | 文本 | 否 | lucide-react 图标名，仅目录和菜单项 |
| 排序 | 数字 | 是 | 数字越小越靠前，默认 100 |
| 状态 | 单选 | 是 | 启用 / 停用，默认启用 |
| 备注 | 文本 | 否 | 说明该节点用途 |

**权限标识命名规范**

```
格式：模块:资源:动作
目录：  admin（无动作部分）
菜单项：admin:users:list、admin:roles:list、admin:audit-logs:list
按钮：  admin:users:create、admin:users:edit、admin:users:delete
        admin:users:reset-password、admin:users:export
        admin:roles:create、admin:roles:edit、admin:roles:delete
        admin:roles:assign-permissions
        admin:audit-logs:export
```

#### 3.4.3 预置菜单树（初始数据）

```

│   ├── 新建角色（按钮）               admin:roles:create
│   ├── 编辑角色（按钮）               admin:roles:edit
│   ├── 停用/启用角色（按钮）          admin:roles:toggle-status

#### 3.4.4 规则说明

- 权限标识全局唯一，重复时阻止保存
- 停用父节点时，其所有子节点同步停用；启用父节点时，子节点状态不自动恢复（需手动启用）
- 已被角色引用的菜单节点不允许删除，需先解绑所有角色后再删除
- 按钮类型节点只能挂载在菜单项下，不能直接挂载在目录下
- 菜单树最多支持 3 级（目录 → 菜单项 → 按钮）
- 菜单排序调整实时生效，无需发布

**验收标准**

- [ ] 菜单列表以树形表格正确展示层级结构
- [ ] 新建菜单时权限标识重复校验正确
- [ ] 停用父节点时子节点同步停用
- [ ] 已被角色引用的节点删除时系统正确拦截
- [ ] 按钮类型节点只能挂载在菜单项下
- [ ] 所有菜单操作写入审计日志

### 3.5 权限配置

**功能描述**

控制不同角色在平台中的访问范围和可操作动作。权限数据来源于菜单管理模块，角色权限配置页直接以菜单树为数据源，不需要单独维护权限列表。

#### 3.5.1 权限模型

平台采用四层 RBAC 模型，菜单管理是第 1-3 层的数据来源：

| 层级 | 数据来源 | 说明 | 示例 |
|------|---------|------|------|
| 模块权限 | 菜单管理（目录节点） | 一级菜单是否可见 | 后台管理、内容中心 |
| 页面权限 | 菜单管理（菜单项节点） | 具体页面是否可访问 | 用户管理页、审计日志页 |
| 操作权限 | 菜单管理（按钮节点） | 具体按钮/动作是否可执行 | 新建、删除、导出 |
| 数据范围 | roles 表 data_scope 字段 | 可见数据的范围（首期预留） | 全量、组织内、项目内 |

**权限计算规则**

- 用户最终权限 = 所有绑定角色权限的并集
- 冲突时以「禁止优先」原则处理
- 首期不设互斥角色约束

#### 3.5.2 权限配置页面

**布局**：左侧菜单树（来自菜单管理，实时加载）+ 右侧操作权限勾选

**交互流程**

1. 进入角色详情 → 点击「配置权限」
2. 左侧展示完整菜单树（目录 / 菜单项 / 按钮三种节点）
3. 勾选目录/菜单项控制页面可见权限，勾选按钮控制操作级权限
4. 勾选父节点自动勾选所有子节点；取消父节点自动取消所有子节点
5. 点击「复制自其他角色」可快速复制已有角色的权限配置
6. 点击「权限预览」可模拟该角色登录后的菜单可见效果
7. 保存后权限立即写入，下次请求实时生效

**底部操作**：保存、重置、复制自其他角色、权限预览

#### 3.5.3 权限执行规则

| 编号 | 规则 |
|------|------|
| P-01 | 未授权菜单在导航中直接隐藏，不展示后提示无权限 |
| P-02 | 用户直接访问无权限页面时，返回「无访问权限」页 |
| P-03 | 无操作权限的按钮不展示或置灰，不允许前端仅隐藏而后端可调用 |
| P-04 | 后端每次请求必须实时校验权限，不依赖登录时缓存的角色快照 |
| P-05 | 权限调整后，下次请求时后端实时拦截生效，不强制退出当前登录用户 |
| P-06 | 用户正在操作的页面在权限撤销后不强制跳转，但操作请求被拦截，返回「权限已变更，请刷新页面后重试」 |
| P-07 | 前端收到权限不足响应后，展示全局提示条引导刷新，不得静默失败 |
| P-08 | 权限变更仅系统管理员可操作 |
| P-09 | 菜单节点停用后，已持有该权限的角色对应权限自动失效，不需要重新配置角色 |

#### 3.5.4 只读权限说明

只读不是独立角色，而是一组权限策略，默认开放以下能力：

**可访问**：工作台、状态清单、内容中心详情查看、周期管理查看、报表中心查看、配置中心查看、帮助中心

**不可操作**：新建、编辑、删除、导入、导出发起、配置修改、审批动作

**验收标准**

- [ ] 不同角色登录后仅可见授权菜单和页面
- [ ] 无权限页面和按钮正确隐藏
- [ ] 后端拦截无权限请求，返回正确错误码
- [ ] 权限调整后下次请求即时生效
- [ ] 只读角色无法执行任何写操作
- [ ] 菜单节点停用后对应权限自动失效
- [ ] 权限配置操作写入审计日志

### 3.6 审计日志

**功能描述**

对平台全模块关键操作进行留痕，用于问题排查、责任认定和合规审计。首期后台管理模块实际写入，其他模块按统一规范预留接入点，开发时同步接入。

#### 3.6.1 日志覆盖范围

| 日志类型 | 说明 | 首期是否写入 |
|----------|------|-------------|
| 用户权限日志 | 用户创建、停用、角色变更、密码重置 | ✅ |
| 角色权限日志 | 角色创建、编辑、停用、权限调整 | ✅ |
| 登录日志 | 登录成功、登录失败、账号锁定 | ✅ |
| 内容编辑日志 | 内容字段、图片、字幕、音轨等编辑行为 | 预留，内容模块开发时接入 |
| 签核日志 | 初始签核、全量签核、解锁 | 预留，签核模块开发时接入 |
| 变更审批日志 | 变更单提交、审批、驳回、取消 | 预留，变更模块开发时接入 |
| 配置修改日志 | 配置档、分类集、媒体配置、路由等修改 | 预留，配置模块开发时接入 |
| 导入日志 | 导入对象、模式、结果、错误明细 | 预留，导入模块开发时接入 |
| 导出日志 | 导出发起、校验结果、状态、下载行为 | 预留，导出模块开发时接入 |

#### 3.6.2 日志列表页

**筛选项**：日志类型、操作人、对象类型、对象名称/ID、时间范围、所属项目

**列表字段**：日志 ID、操作时间、操作人、日志类型、对象类型、对象名称、操作动作、结果、操作

**底部操作**：导出当前筛选结果

#### 3.6.3 日志详情字段

| 字段 | 说明 |
|------|------|
| 日志 ID | 唯一标识 |
| 日志类型 | 内容 / 签核 / 变更 / 配置 / 导入 / 导出 / 用户权限等 |
| 操作人 | 用户名 + 姓名 |
| 操作时间 | 精确到秒 |
| 操作动作 | 新建 / 编辑 / 删除 / 审批 / 导入 / 导出 / 登录等 |
| 对象类型 | 内容 / 周期 / 配置档 / 用户 / 角色等 |
| 对象 ID | 唯一主键 |
| 对象名称 | 便于识别 |
| 变更前 | 字段级前值（JSON） |
| 变更后 | 字段级后值（JSON） |
| 结果 | 成功 / 失败 |
| 失败原因 | 若失败则必填 |
| 来源页面 | 发起来源 |
| 请求编号 | 便于后端追踪 |

#### 3.6.4 日志记录规则

1. 所有写操作必须记录日志，失败操作同样记录
2. 审批类动作必须记录操作人、意见和结果
3. 日志不可编辑、不可删除
4. 默认保留 90 天，到期后自动清理或归档
5. 系统管理员可查看全量日志；其他角色仅可查看授权范围内日志
6. 支持按对象反查，例如「某内容最近 30 天被谁改过」

**验收标准**

- [ ] 后台管理模块所有写操作均生成审计日志
- [ ] 登录成功、失败、账号锁定均生成日志
- [ ] 日志列表支持按操作人、时间、对象类型、结果筛选
- [ ] 日志详情展示字段级变更前后值
- [ ] 日志不可编辑、不可删除
- [ ] 不同角色查看日志范围符合权限矩阵

## 8. 风险与依赖

| 风险/依赖 | 说明 | 应对措施 |
|-----------|------|---------|
| 权限矩阵初始化 | 系统预置角色的默认权限需要在上线前由管理员完成配置，否则业务角色无法正常使用 | 提供权限矩阵建议默认值文档，实施时通过 Seeder 或初始化脚本预设 |
| 其他模块审计日志接入 | 内容、签核、变更、配置等模块需按统一规范写入审计日志 | 输出《审计日志写入规范》文档，各模块开发时同步接入，不允许后补 |
| 数据范围权限 | 首期预留字段但不实现过滤逻辑，如果项目/组织间数据隔离需求紧迫，可能需要提前推进 | 评估各业务角色的实际数据访问范围需求，若有隔离风险提前升级为 P0 |
