# 一、产品功能结构图（正式版）

## 1.1 一级模块结构

建议将地面内容管理平台拆分为 8 个一级模块：

1. 工作台
2. 内容中心
3. 分类与质量
4. 周期与发布
5. 配置中心
6. 报表中心
7. 后台管理
8. 帮助与工具

这样拆分的依据是：

* 你的业务闭环是“内容录入 → 分类挂载 → 状态检查 → 签核 → 校验 → 导出 → 追踪”，因此“分类与质量”“周期与发布”应从内容管理中独立出来。
* 你的产品范围本身已经覆盖工作台、内容中心、分类与状态清单、周期管理、签核与变更管理、导入与批量处理、发布导出、完整性校验、报表中心、配置中心、扩展配置、后台管理和帮助中心，完全具备按业务域重组的基础。

# 二、信息结构图（IA）正式版

## 2.1 顶部一级导航建议

建议最终顶栏为：

* 工作台
* 内容中心
* 分类与质量
* 周期与发布
* 配置中心
* 报表中心
* 后台管理
* 帮助

### 不建议的做法

不建议在顶栏或二级全局导航中直接平铺：

* 视频
* 音频
* 游戏/应用
* 有声书
* 电子书
* 导入
* 组模板
  这类对象型入口。

原因是你的内容对象会持续扩展，且对象类型之间共享同一套详情结构、导入机制、素材管理、批量处理和模板能力；把它们抬到全局高层，会让导航越来越碎，且不利于用户按业务闭环工作。你的 PRD 也明确指出不同内容类型共享文件信息、其他信息、元数据、图片、分类等详情页标签结构。

---

## 2.2 一级模块下的信息结构

### A. 工作台

**入口定位**：默认首页
**页面结构**：

* 顶部：项目/组织、周期、全局搜索
* 左侧：分类树、快速查找
* 右侧：待办、周期里程碑、变更申请、异常提醒

**适用角色**：

* 内容运营
* 审核人员
* 发布人员
* 配置管理员
* 只读用户（只读展示）

这与原 PRD 对工作台的定义一致。

### B. 内容中心

**入口定位**：内容对象主入口
**页面结构**：

* 头部：项目/周期切换、关键字搜索
* 筛选区：内容类型、状态、语言、分级、来源、是否签核
* 主体：内容列表
* 详情：抽屉或详情页，采用 Tab 结构

**建议页内二级页签**：

* 全部内容
* 视频
* 音频
* 游戏/应用
* 有声书/电子书
* 服务类/其他

**说明**：对象类型作为**页内二级页签或筛选器**存在，而不是上升为全局导航。这样既保留操作效率，又不牺牲 IA 的稳定性。内容详情页签应保留文件信息、系列/分集、字幕/音轨、其他信息、元数据、预告素材、图片素材、分类信息等。

### C. 分类与质量

**入口定位**：内容组织和质量门禁中心
**页面结构**：

* 左侧：分类树 / 分类集树
* 右侧：状态清单 / 异常列表 / 校验结果
* 顶部操作：批量挂类、批量处理、创建版本、执行校验

**说明**：这里最适合保留 MMA 风格的“左树右表”结构，因为分类树天然是树形组织，状态清单和异常结果天然是表格展示。原 PRD 也明确要求平台支持左右双栏布局，并支持从分类树进入状态清单。 

### D. 周期与发布

**入口定位**：发布运营主链路
**页面结构建议**：

* 周期列表
* 周期详情
* 签核与变更
* 导出中心
* 导出历史 / 导出差异

**页面流**：
周期列表 → 周期详情 → 签核状态 / 完整性校验 → 导出任务 → 导出历史 / 差异比对

**说明**：这比把“循环/Cycle”做成一个含糊的菜单更清晰，能直接对应你产品里最重要的发布流程。原 PRD 已明确支持签核与变更、周期管理、完整性校验和导出发布的完整闭环。

### E. 配置中心

**入口定位**：发布规则治理中心
**页面结构建议**：

* 配置档列表（主入口）
* 配置详情页

  * 基础配置
  * 对象启用
  * 类型配置
  * 路由 / 分组
  * 偏好设置
  * 扩展配置

**说明**：配置中心不适合像 MMA 一样把 System / Audio / Video / Image 直接散落在导航层级里。更适合以“配置档”为核心对象，再进入配置详情管理对象启用、类型、路由、偏好和扩展项。原 PRD 已经说明配置中心是围绕配置档和对象启用规则来组织的。

### F. 报表中心

**入口定位**：统一分析中心
**页面结构**：

* 顶部页签：媒体报表 / 变更报表 / 质量报表 / 操作报表
* 列表筛选区
* 报表表格
* 导出按钮

**说明**：报表中心应保持统一，不建议分散到周期或内容模块下，避免重复入口。原 PRD 已将报表中心定义为独立一级模块。

### G. 后台管理

**入口定位**：平台治理中心
**页面结构**：

* 左侧菜单：用户管理、角色管理、权限配置、审计日志、通知中心、系统参数
* 右侧：列表页 + 详情/编辑页

**说明**：后台管理应与配置中心严格分离。配置中心管理业务规则，后台管理管理平台治理，这一点在你的角色设计里也已经分开：配置管理员负责配置档、分类集、媒体配置、路由、扩展配置；系统管理员负责用户、角色、权限、后台参数和系统运维能力。

# 四、给你的最终建议

## 可以继承松下的

* 顶部主导航数量控制
* 统一筛选栏 + 搜索 + 面包屑
* 左树右表/右详情的页面骨架

* 把视频/音频/游戏/电子书等内容类型放在全局高位导航
* 把业务流程和对象类型混在同一层级

* 让左侧树承担全站级主导航

# 五、一句话结论

**你们最适合的 IA 不是“参考松下按内容类型导航”，而是“按业务域导航、按对象类型页内切换、按树形场景局部保留左树右表”。** 这和你当前 PRD 的产品目标、业务范围、核心流程、一级导航和通用界面要求是一致的。 

下一步最有价值的是，我帮你继续把这一版整理成**“7. 信息架构 + 8. 各模块功能拆分”的统一编号版**，直接能接在你现有 PRD 后面。
