# 1. 核心元数据架构定义

CMS采用了一个高度可扩展、经过严格验证的元数据架构，专门针对松下机载娱乐系统的数据包规范进行了定制。

## 1.1 业务元数据与技术元数据的合并
我们在 `media` 核心实体中统一管理了业务信息（标题、分类）与技术信息（码率、编码）。

| 核心字段 | 来源描述 |
| :--- | :--- |
| **UID / MID** | 点击 UID 支持在原生弹窗内直接跳转到该媒体的具体详情页 (`/metadata/video`)。 |
| **标题与多语言** | 依赖底层的 `language_lookups` 字典表驱动多语言。 |
| **分类归属** | 通过 `category_media` 将媒体动态挂载到不限深度的左侧树中。 |
| **运行时长** | 从真实的 `.har` 数据清洗提取的真实时长。 |
| **图片规格** | 完全兼容 Phase 6c 暴增的图片规格（如 `_11`, `_22` 规格前缀）。 |

# 2. 目标导向的操作流
CMS UI 围绕标准化、高效率进行设计。目前系统提供了极速的检索与配置交互。

## 2.1 操作流：原生多列全局搜索
**目标：** 实现跨全量数据库的毫秒级检索，并与松下原生的交互风格高度一致。
- **现状**：系统未采用常规的下拉菜单搜索，而是高度复刻了原版的**独立弹窗视图** (`SearchResultsView`)。支持跨标题、UID 等多列进行快速数据定位，大幅提升了操作区域的空间感。

## 2.2 动态布局与专注模式
- **PremiumLayout 显隐逻辑**：系统具备智能的组件判断能力。在涉及核心元数据呈现的地方（如 Status Sheet）自动展现侧边分类树与顶部筛选栏。在进行 Profile 等系统底层级配置时，这些组件将自动隐藏，确保业务人员专注操作。
